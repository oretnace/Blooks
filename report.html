Report

This website was written by Arnold Cantero using Notepad++ with Chrome browser for visual testing.

The following list items were notes I recorded as I was building this website. Please pardon theinformality and improper spelling and grammar at times, I wanted to get ideas and thoughts down as quick as possible. Hopefully you can make sense of what I have written.

BEWARE - MUST HAVEs

  • Code must not be changed for website to work, otherwise syntax errors etc.
  • Locations of all files must not be moved.
  • All pages need in header:
    • meta name viewport
    • main css file link
    • sidebar javascript link
  • Content column must be longer than sidebar - but don't worry, this is already written in the CSS.
  • Need to make sure inputs in form have name if i want to pass them

CHOICES MADE

  • Purpose of the site is to sell books related to the colour blue, just from the idea of wordplay: blue + books. Also because blue is a calm, soothing colour and a personal preference.
  • Written on the go using Notedpad++ and Chrome browser, with the aid of previous lab work and w3schools tutorials. Not sure of any other methods to write the website from scratch, I'd be wasting time if I spent time to research alternative software.
  • Single CSS file as all pages share the same layout and themes. There would be too many files if each page had their own CSS file.
  • Entity name used for copyright (Footer) and arrows symbol (Slide show), just to give it a try. It might be fun.
    • Rounded corners at top of header, bottom of footer, hovers, catalog table, slide show to make the site look smoother.
  • XML Page is renamed 'Catalog', and Order Forms to 'Order Books' so it's easier to understand for the user.
    • In Catalog and Slide show pages, XML is displayed through the HTML output from JavaScript using a for loop through all books in the XML file.
  • In Order Form, method = "post" should be used to protect sensitive data.
    • Required fields are orange as it complements the site's blue colour scheme, grabs user's attention.
  • When selecting country in order form, i didn't want to include the entire list of all countries so i gave user the option to type theirs in.
  • More white space in background in print-friendly version to save ink. Borders.

PAST CHOICES

  • HTML5 semantic tags used:
    • <header>
    • <nav>
    • <main> - for page content
    • <aside> - sidebar
    • <footer>

CHANGES MADE FROM ORIGINAL PLAN

  • Site logo is now an image instead of text. Before, it was a header image under text in <header> which had a modified fill and outline colour in CSS.
  • XML Book data titles are sorted alphabetically - done it manually, not a result of code. So it's easier to find a book by title. I originally wanted the file to allow new books to be appended without having to be in order.
  • Moved JavaScript in some pages from HTML file to a separate .js document: Catalog, Slide show. Script was in HTML code but it looked too long so I separated it.
  • In slide show, added a scroll when text would overflow caption box so all information is retained.
  • Inactive thumbnails stay transparent, active thumbnail turns opaque, lets you know which thumbnail you're looking at.
  • Added whitespace: nowrap to keep two-word nav link names together in smaller screen sizes. The screen width limit has been changed so we don't risk seeing this happening, mobile version screen width limit has been increased.
  • In order form, used <label> tag as I didn't like the appearnce of the text cursor when the cursor hovered over the text box names.
  • XML/Catalog page no longer uses a table to display books as I want the layout to change when the site is in mobile view. When in mobile view, the table would have a fixed width and the contents inside the table wouldn't fit and go off the screen. Instead, they are now in divs so that information next to the book cover moves below it.
  • In form, took out the label tags next to the input texts and used the placeholder attribute instead. Less code for me to write and looks neater in HTML file.
  • Took out required attribute in form, javascript now displays hidden sections depending on selections and makes them required.
  • Took out card billing address details, too much information

CHALLENGES DURING DEVELOPMENT

  • Padding and layout of elements in navigation bar. I just needed to figure out which element I was selecting: <ul>, <li> or <a>.
  • Finding specific books to suit the site's purpose then converting the information to XML was a bit lengthy and time consuming.
  • Struggling to make the right <aside> column fill any gaps and stretch down to the footer.
  • I wasted so much time trying to make a really fancy slide show with JavaScript terms I've never come across before from an example in w3schools when I can use a simpler example from the labs.
  • Writing the JavaScript for the slideshow was a little challenging, I needed to count the number of books and was struggling trying to access variables with a local scope.
  • In order form, Difficulty in trying to get value of selected option from drop down list, wanted to compare it against a value in javascript.
  • Centering elements was a massive pain.
  • In javascript functions, wanted to make code neater by having variable declarations outside of code as the same variables would be used in other variables. However, the triggering event triggers different functions and the declarations arent in the functions.
  • Form doesn't submit how i want to when trying to validate postcode with javascript so used html pattern instead
  • Selecting the right elements in the form with javascript

SUCCESSES

  • Feeling satisfied for incorporating aesthetic improvements like box shadows, hover transitions and slide show thumbnails, looks nicer. Especially how you can select a thumbnail and the active thumbnail stays opaque.
  • When testing on other browsers like Firefox, it was a good thing I used entity names as the £ sign didn't display correctly. But it then displayed correctly when I changed it to the entity name in the HTML code.
  • I have struggled to make the sidebar stretch all the way down to be as long as the content box so I re-wrote the HTML code of the main page layout. The more I messed around with the CSS, the more I got IDs, classes and HTML5 semantic tags confused. Some CSS stylings would work on some elements, using semantic tags actually made styling more confusing for me so I went back to basics and used divs instead. This time around, I successfully made the layout responsive and used less absolute measurements like px. The website now shifts to a mobile view at screen sizes of 500px and less. The page contents are then listed vertically: the navigation links, the content followed by the sidebar, then the footer. I did this by wrapping the content and sidebar divs with another div with overflow auto and had the same background colour as the sidebar to create the illusion. The result was a success after testing the browser window sizes with IE v11, Firefox v50 and Chrome v55.
